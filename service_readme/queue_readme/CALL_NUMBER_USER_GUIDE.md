# 叫号管理界面使用指南

## 🚀 快速开始

### 1. 启动所有服务

确保以下服务已启动：
- ✅ Eureka Server (8761)
- ✅ Gateway Service (8080)
- ✅ Queue Service (8086)
- ✅ Notification Service (8086)
- ✅ Shop Service (8082)
- ✅ User Service (8081)
- ✅ Frontend (3000)
- ✅ Redis (6379)
- ✅ MySQL (3306)

### 2. 访问叫号管理页面

打开浏览器访问：`http://localhost:3000`

使用店员或管理员账号登录：
- **用户名**: `staff` 或 `manager` 或 `admin`
- **密码**: `123456`

登录后，点击左侧菜单：**排队管理** → **叫号管理**

### 核心功能

#### 1. 店铺选择
- 从下拉框选择要管理的店铺
- 切换店铺时自动刷新队列数据
- 默认选择第一个店铺

#### 2. 实时等待队列
显示当前店铺所有等待中的排队记录：
- **排队位置**: #1, #2, #3...（按取号时间排序）
- **排队号码**: 如 A20260519002
- **详细信息**:
  - 用餐人数
  - 排队类型（堂食/外带）
  - 桌位类型（普通桌/卡座/包厢）
  - 联系电话
- **操作按钮**: 📢 叫号

**特殊标识**:
- 第一位排队（#1）会有绿色边框和脉冲动画，提示优先叫号

#### 3. 已叫号队列
显示已叫号但未完成的排队记录：
- **状态徽章**: [已叫号]
- **排队号码**: 如 A20260519001
- **详细信息**:
  - 用餐人数
  - 排队类型
  - 联系电话
  - 叫号时间（格式：HH:MM）
- **操作按钮**: ✔️ 完成

#### 4. 刷新功能
- 点击 "🔄 刷新全部" 按钮手动刷新
- 叫号/完成操作后自动刷新
- 建议每 30-60 秒刷新一次保持数据最新

## 🎯 操作流程

### 场景 1: 顾客取号后等待

1. 顾客在手机上取号
2. 系统生成排队号码（如 A20260519005）
3. 排队记录出现在"实时等待队列"中
4. 顾客手机收到"取号成功"通知

### 场景 2: 店员叫号

1. 店员在叫号管理页面查看等待队列
2. 找到第一位排队（#1 高亮显示）
3. 点击 "📢 叫号" 按钮
4. 确认对话框中核对信息：
   ```
   确定要叫号：A20260519002（2人）吗？
   [确定] [取消]
   ```
5. 点击"确定"
6. 系统执行：
   - ✅ 更新数据库状态（queue_status = 1）
   - ✅ 移动 Redis 队列（waiting → calling）
   - ✅ 推送 WebSocket 通知给顾客
   - ✅ 刷新界面数据
7. 该排队从"等待队列"移到"已叫号队列"
8. 顾客手机收到"叫号通知"弹窗 + 提示音

### 场景 3: 顾客入座完成排队

1. 顾客听到叫号后到店入座
2. 店员在"已叫号队列"中找到对应排队
3. 点击 "✔️ 完成" 按钮
4. 确认对话框：
   ```
   确定完成排队：A20260519001 吗？
   [确定] [取消]
   ```
5. 点击"确定"
6. 系统执行：
   - ✅ 更新数据库状态（queue_status = 2）
   - ✅ 从 Redis 叫号队列移除
   - ✅ 添加到 Redis 完成队列
   - ✅ 推送"排队完成"通知
7. 该排队从"已叫号队列"消失

## 💡 使用技巧

### 1. 高效叫号
- **优先叫第一位**: #1 位置有绿色高亮，表示应该优先叫号
- **批量操作**: 可以快速连续叫号多个排队
- **确认信息**: 叫号前仔细核对人数和电话

### 2. 异常处理
- **顾客过号**: 如果顾客长时间未到，可以标记为"过号"（需要额外功能）
- **取消排队**: 顾客主动取消时，使用"取消排队"功能
- **重新取号**: 过号顾客需要重新取号

### 3. 多店铺管理
- 如果有多个店铺，通过下拉框切换
- 每个店铺的队列独立管理
- 切换店铺时数据自动刷新

### 4. 实时监控
- 观察"等待人数"徽章，了解队列长度
- 高峰期建议每 10-20 秒刷新一次
- 低峰期可以 1-2 分钟刷新一次

## ❓ 常见问题

### Q1: 为什么看不到等待队列？
**A**: 
1. 检查是否选择了正确的店铺
2. 确认该店铺有等待中的排队
3. 点击"刷新全部"按钮
4. 检查网络连接

### Q2: 叫号后顾客没收到通知？
**A**:
1. 确认顾客已连接 WebSocket（登录后查看控制台）
2. 检查 notification-service 是否正常运行
3. 确认顾客的 userId 正确
4. 查看后端日志是否有推送记录

### Q3: Redis 队列和数据库不一致？
**A**:
1. 正常情况下应该一致
2. 如果不一致，可能是 Redis 过期或手动修改了数据库
3. 可以通过"刷新全部"重新同步
4. 严重时可以清空 Redis 队列重启

### Q4: 如何测试叫号功能？
**A**:
1. 先用顾客账号取号（QueueView）
2. 再用店员账号登录叫号管理（CallNumberView）
3. 在两个浏览器窗口中观察实时变化
4. 叫号后查看顾客窗口是否收到通知

## 🛠️ 技术细节

### API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/queue/list` | GET | 获取所有排队记录 |
| `/api/queue/{id}` | GET | 获取单个排队详情 |
| `/api/queue/{id}/call` | PUT | 叫号 |
| `/api/queue/{id}/complete` | PUT | 完成排队 |
| `/api/queue/redis/waiting/{shopId}` | GET | 获取等待队列（Redis） |
| `/api/queue/redis/calling/{shopId}` | GET | 获取叫号队列（Redis） |

### Redis Key 命名

```
queue:waiting:{shopId}     - 等待队列（ZSET）
queue:calling:{shopId}     - 叫号队列（ZSET）
queue:completed:{shopId}   - 完成队列（ZSET）
```

### 数据结构

**Waiting Queue (ZSET)**:
```
Key: queue:waiting:1
Members: [
  { member: "13", score: 1779122400000 },
  { member: "14", score: 1779122460000 },
  { member: "15", score: 1779122520000 }
]
```
